# Design Document: Mail Client System Architecture

## Introduction

This repository serves as a professional-grade template for a modern Python project. It demonstrates a robust, component-based architecture by building the core components for an AI-powered email assistant that interacts with the Gmail API.

## Application Goal

The project emphasizes a strict separation of concerns, dependency injection, and a comprehensive, automated toolchain to enforce code quality and best practices. The system is built on the principle of "programming integrated over time" with component-based design where each component has a single responsibility.

## System Components

The mail client system consists of four main components that work together to provide flexible, distributed mail functionality:

### 1. Mail Client API (`mail_client_api`)
The foundational abstract base class that defines the contract for all mail client implementations:
- Defines the `Client` ABC with abstract methods for mail operations
- Provides the `Message` interface for consistent data structures
- Includes factory functions for dependency injection
- Ensures interface stability across different implementations

### 2. Gmail Client Implementation (`gmail_client_impl`)
A concrete implementation of the `Client` interface that integrates with Google's Gmail API:
- Implements all abstract methods from the `Client` interface
- Handles OAuth2 authentication and token management
- Translates Gmail API responses to standard `Message` objects
- Provides direct access to Gmail functionality

### 3. Mail Client Service (`mail_client_service`)
A FastAPI web service that wraps the Gmail client functionality and exposes it through HTTP endpoints:
- Uses dependency injection to get a mail client instance
- Provides REST endpoints for mail operations (`GET`, `POST`, `DELETE`)
- Handles error mapping to appropriate HTTP status codes
- Uses Pydantic models for request/response serialization
- Includes comprehensive error handling and logging

### 4. Mail Client Adapter (`mail_client_adapter`) 
An HTTP client that implements the `Client` interface by making calls to remote mail services:
- Implements the same `Client` interface as the Gmail client
- Uses an auto-generated HTTP client to communicate with services
- Translates HTTP responses back to domain `Message` objects
- Provides transparent remote access with identical method signatures
- Enables distributed mail client functionality

### Data Models

The service uses two Pydantic models defined in `main.py`:

**MessageSummary** - for listing messages:
- `id: str` - message identifier  
- `from_: str | None` - sender email (optional)
- `subject: str | None` - message subject (optional)
- `date: str | None` - message date (optional)

**MessageDetail** - for full message content:
- `id: str` - message identifier
- `from_: str` - sender email
- `to: str` - recipient email  
- `date: str` - message date
- `subject: str` - message subject
- `body: str` - message body content

## Service Implementation

### FastAPI Endpoints

The service implements four endpoints that mirror the `Client` interface:

1. **`GET /messages`** - `get_messages_summary()`
   - Returns list of `MessageSummary` objects
   - Accepts `max_results` query parameter (default: 10)
   - Maps exceptions to HTTP 500 errors

2. **`GET /messages/{message_id}`** - `get_message_detail()`  
   - Returns `MessageDetail` object
   - Maps exceptions to HTTP 404 errors

3. **`POST /messages/{message_id}/mark-as-read`** - `mark_message_as_read()`
   - Returns success message or HTTP 500 on failure
   - Uses boolean return from client method

4. **`DELETE /messages/{message_id}`** - `delete_message()`
   - Returns success message or HTTP 500 on failure  
   - Uses boolean return from client method

### Dependency Injection

The service uses FastAPI's dependency injection through `get_mail_client()`:
- Calls `get_client(interactive=False)` to get a client instance
- Maps `RuntimeError` to HTTP 503 for authentication failures
- Uses `MailClientDep` type alias for endpoint parameters

## Adapter Implementation

### ServiceClientAdapter Class

The adapter implements the `Client` interface using an auto-generated HTTP client:

```python
class ServiceClientAdapter(Client):
    def __init__(self, base_url: str):
        self._client = AutoGeneratedClient(base_url=base_url)
```

**Methods implemented:**
- `get_messages()` - calls `_client.messages.list_messages.sync()` and yields `ServiceMessage` objects
- `get_message()` - calls `_client.messages.get_message.sync()` and returns `ServiceMessage`  
- `delete_message()` - calls `_client.messages.delete_message.sync()` with exception handling
- `mark_as_read()` - calls `_client.messages.mark_as_read.sync()` with exception handling

### ServiceMessage Class

Wraps HTTP response objects to implement the `Message` interface:

```python
class ServiceMessage(BaseMessage):
    def __init__(self, detail: MessageDetail | MessageSummary):
        self._detail = detail
```

**Properties:** Uses `getattr()` with empty string defaults to safely access:
- `id`, `from_`, `to`, `date`, `subject`, `body`

### Error Handling

The adapter uses try/catch blocks for operations:
- `delete_message()` and `mark_as_read()` return `False` on any exception
- `get_messages()` and `get_message()` let exceptions propagate

## Testing Strategy

### What We Tested

We tested three main components and their interactions:

1. **Mail Client Service**: FastAPI endpoints, dependency injection, error handling, Pydantic models
2. **Mail Client Adapter**: Interface compliance, auto-generated client integration, error translation
3. **Integration Flow**: Gmail client dependency injection and authentication

### Test Types

**Unit Tests** (Component-level):
- Service endpoint logic with mocked Gmail client using FastAPI TestClient
- Pydantic model validation and serialization
- Dependency injection function testing
- Error handling for various failure scenarios

**Integration Tests** (Component interaction):
- Gmail client authentication and API connectivity
- Dependency injection system verification
- Factory function integration testing

**End-to-End Tests** (Full system):
- Complete user workflow simulation (planned but not fully implemented)
- Live Gmail API integration with credential management
- Service startup and health checking

### Mocking Strategy

**Service Tests**: Use FastAPI dependency overrides to inject mock Gmail clients, avoiding real Gmail API calls. The test fixture creates a TestClient with the dependency override set to return a mock client, then clears the override after testing.

**Integration Tests**: Mock Gmail client initialization for authentication testing by patching the get_client factory function to return a mock Gmail client, allowing tests to verify dependency injection and factory functions work correctly.

**Adapter Tests**: Mock the auto-generated client components by patching the adapter's internal client object and setting return values for the HTTP method calls, enabling testing of adapter logic without making real HTTP requests.

### Interface Compliance

**Testing Approach**: We verified adapter compliance through multiple strategies:

1. **Type Checking**: Adapter inherits from abstract `Client` class, ensuring method signatures match
2. **Behavioral Testing**: Integration tests verify adapter methods return same types as original interface
3. **Contract Testing**: Tests verify adapter handles all interface methods with correct return types and error behaviors

**Compliance Verification**: Type checking ensures the adapter inherits from the Client abstract base class, which guarantees method signatures match the original interface. Integration tests verify dependency injection works by checking that the factory returns a GmailClient instance and that it has all required methods (get_messages, get_message, delete_message, mark_as_read). Service tests verify interface compliance through FastAPI TestClient by making HTTP requests and asserting that the underlying mock client methods are called with correct parameters.

The adapter successfully maintains interface compliance by implementing all abstract methods from the `Client` base class and ensuring return types and behaviors match the original contract.
